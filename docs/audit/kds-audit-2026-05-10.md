# KDS Audit - 2026-05-10

Scope:
- `src/app/[locale]/dashboard/kds/**`
- `src/components/kds/**`
- Server actions, RPCs, realtime subscriptions, and migrations touching `order_item_station_status`

Notes:
- Read-only audit. No application code was changed.
- Source-backed findings only; no live Supabase SQL was executed in this pass.
- No direct cross-branch leak was found in the normal KDS page/action path after the current branch filters: initial page load filters by user branch, realtime filters `order_item_station_status` by branch when a branch is known, and server actions check same-branch ownership for non-global users. Direct database/RPC role gaps remain below.
- Realtime/polling cleanup is present: the board removes the Supabase channel on unmount, clears polling on reconnect/effect cleanup, clears the clock interval, and cards clear undo timers.
- Current migration `089_kds_rls_hardening.sql` enforces `UNIQUE(item_id)`, so one item cannot be assigned to multiple stations at the database constraint level if that migration is deployed.

## Top 5 Critical Fixes

1. Lock down `order_item_station_status` RLS and the `update_order_item_station_status` RPC to KDS-authorized roles only, not every same-branch authenticated role.
2. Align the station taxonomy end to end. The requested five-station model cannot be guaranteed while UI constants, DB enums, seed data, and trigger mapping use different station names.
3. Replace silent fallback routing (`ELSE 'main'`) with an explicit unassigned/error path so new menu items cannot disappear into the wrong station.
4. Make station status updates transactional and conflict-aware with server-side transition validation and expected-current-status checks.
5. Harden station bumping so it can only complete rows that are actually ready, verifies affected row counts, and reports a visible operator error on failure.
